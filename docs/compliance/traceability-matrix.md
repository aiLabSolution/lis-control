# Traceability Matrix (Seed) — Requirement → Verification → Evidence

> **Stage-0 SEED — not the finished dossier.** This is a scaffold prepared for **LIS-10 / S0.8** ("[S0.8] Compliance scaffold"), drafted by an agent and **pending human review** (date: **2026-06-23**). It is the *authoritative registry* of `REQ-*` IDs for the LabSolution LIS programme: the other three Stage-0 compliance artifacts (Validation Master Plan outline, NPC checklist, threat model) are checked for consistency against *this* table, and **this table is the single source of truth for each requirement's status** — sibling docs reference the status here rather than restating it. It is seeded at Stage 0 and **maintained continuously through Stage 5**, where it becomes the spine of the signed ISO 15189:2022 IQ/OQ/PQ validation dossier (per `LIS_IMPLEMENTATION_PLAN.md` §1 — "the verification column is the contract").

## 1. Purpose, scope, and how this grows

**Purpose.** Maintain a single line-of-sight from every regulatory/quality **requirement** → the **verification level + test/mechanism** that demonstrates it → the **issue** that produces the work → the **evidence artifact** that an auditor can inspect. Under ADR-0001 the umbrella `lis-control` pins every code component (OpenELIS core at `core/openelis`, future `edge/drivers`, `plugins`, `deploy/kit`, `infra`) at a known SHA, so **one umbrella commit = one reproducible, pinned snapshot of the whole system**. That snapshot *is* the IQ baseline; this matrix records what each later snapshot must verify as a **delta on a known base** — never a black-box revalidation.

**Scope (Stage 0 SEED).** Every requirement in the canonical `REQ-*` seed is registered below, plus additions made with the same prefixes (`REQ-PRIV-08`, `REQ-AUD-03`, `REQ-SEC-05`, `REQ-VAL-03`, `REQ-QMS-04`, `REQ-CONF-02`, `REQ-LIC-02`). No row is `[DONE]` yet — Stage 0 is foundations; the Stage-0 sibling issues (LIS-3..LIS-9) are still in flight and their evidence rows close as those issues land.

**How it grows.**
- **Each stage adds rows and closes rows.** As LIS-3..LIS-9 merge, their `[PLANNED-S0]` rows attach concrete evidence (CI run URLs, signed reports, schema migrations pinned at the umbrella SHA) and flip toward `[DONE]`.
- **Every new requirement gets a `REQ-<PREFIX>-NN` ID here first**, then is referenced from the VMP, NPC checklist, and threat model — not the other way around. The seven Stage-0 additions above carry a **"Referenced in" pointer** (§2 "Linked issue" / prose) showing where each is back-propagated into the owning artifact; a reviewer can confirm the registry-first invariant by following those pointers.
- **Stage 5 executes the matrix end-to-end:** each row's test is run, signed, and its evidence frozen at a specific umbrella commit; deviations are logged and closed against the row.

**Column legend.**
- **Req ID** — canonical `REQ-<PREFIX>-NN`; stable, never recycled.
- **Source** — governing regulation/standard, by the authoritative names (RA 10173, NPC Circular 2022-04, RA 4688, DOH AO 2021-0037, RA 5527, ISO 15189:2022, ADR-0001).
- **Workstream** — **A** Core (OpenELIS fork) · **B** Edge (driver/interface layer, MLLP/ASTM) · **C** API (FHIR R4, offline sync) · **D** Compliance/QA.
- **Verification level** — pyramid L1–L6 (`LIS_IMPLEMENTATION_PLAN.md` §1; `diagrams/08-verification-pyramid.png`): L1 Unit · L2 Component · L3 Bench conformance · L4 Integration E2E · L5 Resilience/chaos · L6 Validation/regulatory.
- **Test / mechanism** — the concrete check that produces evidence.
- **Linked issue** — Stage-0 sibling (LIS-n / S0.x) or the later Stage that owns the work.
- **Evidence artifact** — what an auditor inspects (in-repo path under the umbrella, CI run, or signed report), pinned at an umbrella SHA.
- **Status** — `[DONE]` (verified, evidence frozen — *none yet*) · `[PLANNED-Sx]` (planned in stage x; `Sx` is the stage of **first verification**) · `[NEEDS-HUMAN]` (blocked on a human decision, appointment, signature, or external filing). Where a row is `[PLANNED-Sx]` but its completion depends on an undecided human input, a **blocked-by** note names the gating deferred decision.

## Deferred decisions (HITL)

- **Regulatory ownership (Open Decision #5).** No `REQ-PRIV-*` / `REQ-VAL-01` row can be *owned* until leadership names who owns NPC registration, the ISO 15189 validation dossier, and per-customer lab-licensing alignment. This directly blocks LIS-10 execution and gates DPO appointment (REQ-PRIV-06).
- **Deployment topology (Open Decision #3).** Central-cloud-thin-site vs full on-prem-per-site materially changes REQ-PRIV-08 (data residency), REQ-SEC-01/02 (where TLS and at-rest encryption apply), and the RES-* sync rows. The matrix cannot finalize those rows' mechanisms until topology is fixed.
- **Interface engine: build vs buy (Open Decision #6).** Open Integration Engine (MPL-2.0) vs bespoke drivers vs reusing openelis-analyzer-bridge is the validated object behind REQ-SEC-03 (channel isolation), REQ-CONF-01/02 (what must be bench-conformance-validated), and the entire Workstream-B edge. The choice directly drives the license inventory (REQ-LIC-01/02 — adopting the analyzer-bridge is conditioned on confirming its license / HOLD-001) and the threat surface (threat model TB-1/TB-2). Adding/swapping the engine is also a change-control delta (REQ-QMS-03).
- **At-rest key-management root-of-trust (gates REQ-SEC-02).** REQ-SEC-02 cannot move past `[NEEDS-HUMAN]` to a designable spec until key custody / root-of-trust is decided (cross-ref threat model §8 / R2). It is not merely a Stage-5 verification task when its foundation is undecided.
- **Exact ISO 15189:2022 and NPC clause numbers** are marked "(confirm exact clause)" throughout — a human with the standard text and current NPC issuances must confirm before the dossier is signed.
- **HOLD-001 / REQ-LIC-02.** The openelis-analyzer-bridge license is "TBD"; reuse is blocked and the row stays `[NEEDS-HUMAN]` until a license is confirmed in writing. Its disposition also depends on Decision #6 above.
- **DOH AO 2021-0037 currency.** A draft amendment is in HFSRB public consultation (not yet signed); REQ rows citing it must be re-confirmed before go-live.
